# Кеширование данных

> **ID:** LOGIC-009
> **Тип:** Архитектурный паттерн / Стратегия
> **Приоритет:** P1
> **Связанные требования:** NFR-033, NFR-048, NFR-049; NFR-006

---

## 1. Назначение

Стратегия кеширования данных на клиентском устройстве для обеспечения производительности, снижения нагрузки на API и улучшения пользовательского опыта при нестабильном интернет-соединении.

**Сценарии:**
- Кеширование расписания для быстрого отображения
- Кеширование профиля пользователя
- Кеширование настроек уведомлений
- Офлайн-доступ к последним данным
- Инвалидация кеша при обновлении данных

---

## 2. Общая стратегия кеширования

### 2.1. Принципы

| Принцип | Описание |
|---------|----------|
| **Источник истины** | Бэкенд всегда является источником истины (R-004) |
| **Приоритет ответа** | Приоритет отдаётся ответу API над кешем |
| **Актуальность** | Данные в кеше имеют TTL (Time-To-Live) |
| **Прозрачность** | Пользователь видит индикацию загрузки, даже если данные есть в кеше |
| **Инвалидация** | Кеш инвалидируется при изменении данных |

### 2.2. Типы кеширования

| Тип | Описание | Использование |
|-----|----------|---------------|
| **In-memory** | Хранение в оперативной памяти | Быстрый доступ, не сохраняется между сессиями |
| **Persistent** | Хранение в файловой системе | Сохраняется между сессиями |
| **Encrypted** | Хранение в безопасном хранилище | Конфиденциальные данные (токены) |

---

## 3. TTL по типам данных

| Тип данных | TTL | Стратегия обновления |
|------------|-----|---------------------|
| **Расписание (слоты)** | 5 минут | При каждом открытии экрана |
| **Профиль пользователя** | 1 час | При открытии экрана профиля |
| **Мои записи** | 5 минут | При каждом открытии экрана |
| **Настройки уведомлений** | 1 час | При открытии экрана настроек |
| **Абонемент** | 5 минут | При каждом открытии экрана |
| **Оценки инструкторов** | 1 час | При открытии экрана оценок |
| **Системные параметры** | 1 час | При запуске приложения |
| **Изображения (аватары)** | 24 часа | При загрузке изображения |

## 5. Стратегии обновления кеша

### 5.1. Cache-First (сначала кеш)

| Элемент | Описание |
|---------|----------|
| **Применение** | Изображения, статичные данные |
| **Поток** | Сначала показать кеш, затем обновить в фоне |
| **Преимущество** | Быстрое отображение |
| **Недостаток** | Может показывать устаревшие данные |

### 5.2. Network-First (сначала сеть)

| Элемент | Описание |
|---------|----------|
| **Применение** | Расписание, записи, критичные данные |
| **Поток** | Сначала запрос к API, при ошибке — кеш |
| **Преимущество** | Всегда актуальные данные |
| **Недостаток** | Медленнее при плохом интернете |

### 5.3. Стратегия по типам данных

| Тип данных | Стратегия | Обоснование |
|------------|-----------|-------------|
| Расписание | Network-First | Критично для записи, должно быть актуальным |
| Профиль | Network-First | Данные могут измениться |
| Мои записи | Network-First | Статусы записей могут меняться |
| Абонемент | Network-First | Остаток занятий критичен |
| Настройки | Network-First | Могут быть изменены на сервере |
| Изображения | Cache-First | Не критичны, кешируются надолго |
| Системные параметры | Network-First | Могут быть изменены администратором |

## 6. Инвалидация кеша

### 6.1. События инвалидации

| Событие | Действие |
|---------|----------|
| **Истечение TTL** | Автоматическая инвалидация |
| **Изменение данных на сервере** | Инвалидация через push-уведомление |
| **Успешное API-действие** | Инвалидация связанных данных |
| **Выход из аккаунта** | Полная очистка кеша |
| **Смена аккаунта** | Полная очистка кеша |

### 6.2. Инвалидация после API-действий

| Действие API | Инвалидируемые данные |
|--------------|----------------------|
| Создание записи (POST /bookings) | Расписание, Мои записи, Абонемент |
| Отмена записи (POST /bookings/{id}/cancel) | Мои записи, Расписание |
| Изменение проката (PATCH /bookings/{id}/rental) | Мои записи |
| Обновление профиля (PATCH /profile) | Профиль |
| Обновление настроек (PATCH /notifications/settings) | Настройки уведомлений |
| Создание оценки (POST /ratings) | Оценки инструкторов |

### 6.3. Инвалидация по push-уведомлениям

| Тип push | Инвалидируемые данные |
|----------|----------------------|
| Отмена скалодромом | Расписание, Мои записи |
| Изменение времени | Мои записи, Расписание |
| Смена инструктора | Мои записи, Расписание |
| Напоминание об оценке | — (только уведомление) |

## 7. Структура кеша

### 7.1. Ключи кеша

| Тип данных | Ключ | Формат значения |
|------------|------|-----------------|
| Расписание | `slots_{date_from}_{date_to}_{format}` | JSON массив слотов |
| Расписание (день) | `slots_{date}` | JSON массив слотов |
| Профиль | `profile_{clientId}` | JSON профиля |
| Мои записи | `bookings_{clientId}_{status}` | JSON массив записей |
| Абонемент | `subscription_{clientId}` | JSON абонемента |
| Настройки | `notification_settings_{clientId}` | JSON настроек |
| Системные параметры | `system_parameters` | JSON параметров |
| Изображения | `image_{url_hash}` | Байты изображения |

### 7.2. Метаданные кеша

| Поле | Тип | Описание |
|------|-----|----------|
| `data` | any | Собственно данные |
| `cachedAt` | timestamp | Время сохранения в кеш |
| `ttl` | seconds | Время жизни в секундах |
| `version` | string | Версия данных (для миграций) |

## 9. Политика повторных попыток

### 9.1. Retry-стратегия для API

| Параметр | Значение |
|----------|----------|
| **Максимум попыток** | 3 |
| **Начальная задержка** | 1 секунда |
| **Множитель** | 2 (экспоненциальная задержка) |
| **Таймаут запроса** | 10 секунд |

### 9.2. Применение retry

| Тип запроса | Retry | Обоснование |
|-------------|-------|-------------|
| GET (чтение) | Да | Идемпотентный, безопасно повторять |
| POST (создание) | Нет | Может создать дубликат |
| PUT/PATCH (обновление) | Нет | Может привести к конфликтам |
| DELETE (удаление) | Нет | Может привести к ошибкам |

---

## 10. Критерии приёмки

1. Данные кешируются согласно стратегии для каждого типа данных
2. TTL для каждого типа данных соответствует спецификации
3. Приоритет отдаётся ответу API над кешем
4. При ошибке API используются кешированные данные (если есть)
5. При отсутствии кеша и ошибке API показывается ошибка
6. Кеш инвалидируется при истечении TTL
7. Кеш инвалидируется после успешных API-действий
8. Кеш инвалидируется по push-уведомлениям
9. При отсутствии интернета показывается офлайн-баннер
10. Повторные попытки запросов работают согласно политике
