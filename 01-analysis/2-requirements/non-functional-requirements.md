# Нефункциональные требования (НФТ) — Клиентское приложение «Вертикаль»

> **Назначение:** Обеспечить качественные характеристики приложения, которые не связаны напрямую с конкретными функциями, но критичны для успешного использования в реальных условиях скалодрома.

---

## Легенда приоритетов

| Приоритет | Смысл |
|-----------|--------|
| **P0 — Must** | Критично для запуска сезона; без этого приложение непригодно к использованию |
| **P1 — Should** | Важно для качества UX, но можно запустить с минимальной реализацией |
| **P2 — Could** | Желательно, но откладывается на следующие версии |

## 1. Производительность и отзывчивость

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-001** | Время отображения экрана расписания после запуска приложения не должно превышать **2 секунд** при стабильном интернет-соединении (4G/Wi-Fi). | P0 | Клиенты открывают приложение в зале или перед выходом — долгая загрузка вызывает раздражение и отказ от использования |
| **NFR-002** | Подтверждение записи на тренировку (нажатие «Записаться» → получение ответа API) должно занимать не более **1.5 секунд**. | P0 | В часы пик слоты разбираются за секунды; медленная запись = потеря клиента |
| **NFR-003** | Приложение должно корректно работать при **нестабильном интернет-соединении** (в ангаре возможны перебои): повторные попытки запросов с экспоненциальной задержкой (retry policy) не менее **3 попыток** с таймаутом **10 секунд**. | P0 | Скалодром находится в бывшем складском ангаре — качество связи может быть нестабильным |
| **NFR-004** | Время запуска приложения (cold start) не должно превышать **3 секунд** на актуальных устройствах (Android 10+). | P1 | Быстрый запуск важен для сценариев «записаться на месте» |
| **NFR-005** | Приложение должно поддерживать **плавную прокрутку** списка слотов (60 fps) без рывков при загрузке данных. | P1 | Влияет на восприятие качества приложения |

## 2. Надёжность и отказоустойчивость

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-006** | Приложение должно корректно обрабатывать **сценарии отсутствия сети** (offline): показывать понятное сообщение, а не «белый экран» или краш. | P0 | Ангар = возможные проблемы с интернетом; клиент должен понимать, что происходит |
| **NFR-007** | Система должна корректно обрабатывать **ошибки API** (4xx, 5xx) с понятными пользовательскими сообщениями на русском языке. | P0 | Бэкенд — black box; важно, чтобы клиент не пугался технических ошибок |
| **NFR-008** | В случае сбоя API при записи приложение **не должно создавать дублирующиеся брони** (идемпотентность на стороне клиента: повторный запрос с тем же ID не создаёт вторую запись). | P0 | Критично для бизнеса: двойная бронь = конфликт на тренировке |
| **NFR-009** | Приложение должно корректно отображать **empty state** и **loading state** для всех экранов с данными (расписание, мои записи, профиль). | P0 | Улучшает пользовательский опыт и снижает тревожность при отсутствии данных |
| **NFR-010** | Crash rate приложения не должен превышать **1%** от общего числа сессий в первые 2 недели после релиза. | P0 | Стабильность — основа доверия к приложению |
| **NFR-011** | Приложение должно сохранять **последнее состояние** при случайном закрытии (например, входящий звонок) и восстанавливать его без потери данных. | P1 | Клиенты часто отвлекаются в зале; потеря введённых данных раздражает |

## 3. Безопасность

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-012** | Аутентификация должна выполняться через **защищённый канал** (HTTPS/TLS 1.2+); передача паролей/телефонов в открытом виде запрещена. | P0 | Передаются персональные данные клиентов (телефон, email, ФИО) |
| **NFR-013** | Токен сессии должен храниться в **безопасном хранилище** устройства (Android Keystore / EncryptedSharedPreferences). | P0 | Предотвращает кражу сессии при компрометации устройства |
| **NFR-014** | Приложение не должно хранить **платёжные данные** клиентов локально (вне MVP онлайн-оплата отсутствует; подготовка к будущей интеграции не должна нарушать безопасность). | P0 | Даже в MVP не должно быть риска утечки платёжной информации |
| **NFR-015** | Все запросы к API должны содержать валидный токен авторизации; при его истечении приложение должно выполнять **бесшовное обновление** (refresh token) без потери текущего контекста. | P0 | Клиент не должен перелогиниваться во время записи |
| **NFR-016** | Приложение должно запрашивать разрешения (push-уведомления, камера для фото профиля — если реализована) в **контексте** — объясняя, зачем нужно разрешение, перед запросом. | P1 | Прозрачность повышает доверие и процент согласий |

## 4. Использование и доступность

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-017** | Все интерактивные элементы (кнопки, ссылки) должны иметь размер **не менее 44×44 pt** для удобства нажатия пальцем (Mobile Accessibility Guidelines). | P0 | Пользователи взаимодействуют с приложением в зале, часто на ходу — мелкие элементы затрудняют использование |
| **NFR-018** | Критические действия (запись, отмена, удаление) должны иметь **подтверждение** (диалог с явным действием), чтобы избежать случайных нажатий. | P0 | Отмена записи по ошибке = потеря места; запись по ошибке = перегрузка расписания |
| **NFR-019** | Приложение должно поддерживать **системный шрифт** и адаптироваться к настройкам размера текста в ОС (Dynamic Type / Accessibility). | P1 | Учитывает потребности клиентов с ослабленным зрением |
| **NFR-020** | Основные сценарии (просмотр расписания, запись, мои записи) должны быть доступны **не более чем за 3 касания** от главного экрана. | P0 | Клиенты должны быстро записываться, не блуждая по меню (особенно в часы пик) |
| **NFR-021** | Все сообщения об ошибках должны быть на **русском языке**, понятны не-техническому пользователю и предлагать путь решения (где возможно). | P0 | Целевая аудитория — широкая, включая родителей детей и людей без IT-опыта |
| **NFR-022** | Приложение должно иметь **тёмную тему** (опционально, но желательно) для использования в помещении с разным освещением. | P2 | Ангар может иметь неравномерное освещение; тёмная тема снижает нагрузку на глаза |

## 5. Совместимость и платформа

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-023** | Приложение должно поддерживать **Android 10 (API 29) и выше** (минимальная версия). | P0 | Охватывает ~90% актуальных устройств на рынке; зафиксировано в `domain-description.md` |
| **NFR-024** | Приложение должно корректно отображаться на экранах **от 4.7" до 6.8"** с соотношениями 16:9 и 20:9. | P0 | Разнообразие устройств у клиентов |
| **NFR-025** | Поддержка **iOS** — вне MVP (P2); однако архитектура должна допускать портирование без полной переработки (чистая бизнес-логика, отдельный UI). | P2 | Зафиксировано в `domain-description.md` §8 |
| **NFR-026** | Приложение должно корректно работать на **разных версиях Android WebView** (если используется встроенный браузер для отображения контента). | P1 | Возможные интеграции с Telegram/Email через WebView |

## 6. Уведомления (Push)

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-027** | Push-уведомления должны доставляться в течение **30 секунд** с момента события на сервере. | P0 | Отмена тренировки скалодромом требует оперативного информирования клиента |
| **NFR-028** | Приложение должно поддерживать **глубокие ссылки** (deep linking) из push-уведомлений: «Записаться на другой слот» → открывает экран расписания. | P0 | Зафиксировано в UC-015 и US-081 |
| **NFR-029** | Приложение должно корректно обрабатывать нажатие на push при **закрытом и фоновом** состоянии. | P0 | Клиент может получить уведомление в любой момент |
| **NFR-030** | Клиент должен иметь возможность **отключить push-уведомления** (кроме критических отмен скалодромом?) — через настройки ОС. | P1 | Соответствует FR-087 (настройка времени напоминания) |
| **NFR-031** | При отсутствии FCM-токена (Google Play Services) уведомления не должны приводить к крашу; должен использоваться fallback (если применимо). | P1 | Устройства без Google Services (например, Huawei) — потенциальная аудитория |

## 7. Масштабируемость и нагрузка

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-032** | Приложение должно поддерживать одновременную работу **до 200 клиентов** в часы пик (запись на популярные слоты). | P0 | Скалодром может иметь пиковые нагрузки (вечер, выходные) |
| **NFR-033** | Кеширование данных (список слотов, профиль) должно уменьшать количество запросов к API при частом открытии приложения (TTL — **5 минут** для расписания, **1 час** для профиля). | P1 | Снижает нагрузку на бэкенд и ускоряет отображение |
| **NFR-034** | Приложение должно корректно работать при **пакетных обновлениях** расписания (например, публикация всей недели сразу) без «зависаний» UI. | P1 | Расписание может публиковаться блоками |

## 8. Тестируемость и мониторинг

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-035** | Приложение должно логировать критические события (ошибки API, crash, успешная запись, отмена) с возможностью выгрузки в удалённую систему (Firebase Crashlytics / аналоги). | P0 | Быстрое обнаружение и исправление проблем после релиза |
| **NFR-036** | В логах **не должны** фигурировать персональные данные (телефон, email, ФИО) в открытом виде (маскировка). | P0 | Соответствие законодательству о персональных данных |
| **NFR-037** | Приложение должно поддерживать **интеграцию с аналитикой** (события: открытие экрана, запись, отмена) для оценки пользовательского поведения (анонимно). | P1 | Помогает Оле понимать популярность форматов и загруженность |
| **NFR-038** | Приложение должно иметь **режим тестирования** (staging-среда) для проверки новых версий без влияния на реальных клиентов. | P0 | Без этого невозможно безопасное обновление |

## 9. Энергопотребление и трафик

| ID | Формулировка | Приоритет | Обоснование |
|----|--------------|-----------|-------------|
| **NFR-039** | Приложение не должно разряжать батарею более чем на **5% за час** активного использования (фоновые процессы минимизированы). | P1 | Клиенты используют телефон в зале; быстрое разряжение вызывает негатив |
| **NFR-040** | Ежемесячное потребление трафика не должно превышать **100 МБ** на пользователя (без учёта изображений инструкторов, если они загружаются). | P1 | Не у всех клиентов безлимитный интернет |
| **NFR-041** | Изображения (аватарки, фото инструкторов) должны кешироваться и загружаться в **оптимизированном формате** (WebP / сжатие). | P1 | Снижает трафик и ускоряет загрузку экранов |

## Примечания для команды разработки

1. **Приоритизация:** НФТ с приоритетом **P0 (36 требований)** должны быть выполнены к релизу MVP. В случае конфликта с FR — FR имеют бизнес-приоритет, но НФТ P0 не должны блокировать ключевые сценарии.

2. **Тестирование:** Особое внимание — тестам на **нестабильное соединение** и **ошибки API**, так как бэкенд — black box и поведение при сбоях критично.

3. **Мониторинг:** Рекомендуется внедрить **Firebase Crashlytics + Analytics** с первого релиза для сбора данных об ошибках и пользовательском поведении.

4. **Архитектура:** Чистое разделение UI/Business/Data слоёв (например, MVVM + Clean Architecture) облегчит поддержку и будущий порт на iOS.
