# Бизнес-требования

> **Скоуп — клиентское мобильное приложение и API для него.** Бизнес-цели, обслуживаемые
> существующей инфраструктурой (интерфейс инструктора, админка владельца, формирование расписания),
> в этот список не входят. Текущая поставка фокусируется на клиентском сценарии, который снимает
> основную боль — хаос записи и двойные брони.

> **О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для
> трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее
> заведённых требований вынесена в развитие/Phase 2). Нумерация намеренно не нормализуется,
> чтобы не ломать связи в таблице-источнике и смежных документах.

| M-6 | Готовность к старту сезона. | Запуск в срок ~2 месяцев с критичным набором функций (просмотр слотов, запись, отмена, абонемент, уведомления) | BR-7 |
| M-7 | Вовлечённость клиентов в оценку инструкторов (доля клиентов, оставивших оценку после тренировки). | ≥ 30% в первые 3 месяца после внедрения (Phase 2) | BR-13 |
| M-8 | Удовлетворённость клиентов качеством приложения (оценка в магазине приложений). | Средний рейтинг ≥ 4.2 из 5 в первые 3 месяца | BR-7, BR-9, BR-12 |

## Ограничения верхнего уровня

### 1. Срок и бюджет
- **Жёсткая цель** — начало нового сезона (~2 месяца с момента утверждения требований).
- **Бюджет ограничен** — строго MVP-подход, «навороты» откладываются на Phase 2.
- **Источник:** Бриф (письмо Оли); BR-7.

### 2. Платформа
- **Только Android** на старте (iOS — Phase 2).
- **Русский язык** интерфейса (многоязычность — Phase 2).
- **Офлайн-режим** — вне MVP (Phase 2).
- **Источник:** Описание домена → «Платформа и локализация (MVP)».

### 3. Оплата
- **Онлайн-оплата** вне скоупа MVP.
- Оплата производится **на месте** (наличные / перевод на карту).
- Приложение отображает цены, но не обрабатывает платежи.
- **Источник:** Бриф (письмо Оли); BR-8; FR-091.

### 4. Интеграция с бэкендом
- Существующий бэкенд — **black-box источник истины**.
- Клиентское приложение полагается на **контракт API**.
- Гарантия отсутствия двойных броней — **на стороне сервера** (атомарная проверка мест).
- Детали серверной реализации — **вне скоупа клиентского приложения**.
- **Источник:** Бриф (R-004); Описание домена → «Интеграционная модель».

### 5. Роли и интерфейсы
- **Только клиентская роль** в текущей поставке.
- Интерфейсы инструктора и администратора/владельца уже существуют в инфраструктуре.
- **Источник:** Бриф (R-028); Описание домена → «Что входит в поставку».

### 6. Данные
- **Нет исторических данных** и легаси-схем.
- Каноническая схема данных — **контракт API**.
- Бэкенд отдаёт все поля контракта (прокатный тариф, зона/формат, свободный фонд снаряжения).
- **Источник:** Бриф (R-015).

### 7. Функциональные ограничения MVP
| Что входит в MVP | Что отложено (Phase 2) |
| :-- | :-- |
| Регистрация по телефону, Telegram, email | iOS |
| Просмотр расписания на 7 дней | Онлайн-оплата |
| Запись на тренировку с выбором проката | Программа лояльности |
| Отмена записи с порогом `n` минут | Лист ожидания |
| Управление абонементом (пакетом занятий) | Офлайн-режим |
| Push-уведомления (напоминания, отмена скалодромом, изменения) | Многоязычность |
| Блокировка записи при долге | Расширенная аналитика |
| Отображение цен (без онлайн-оплаты) | Фильтр по инструктору |
| Оценка инструкторов (звёзды + комментарий) | — **уточнено:** оценка инструкторов включена в MVP (см. BR-13) |
| Отображение статусов записей | — |
| Email-дублирование уведомлений | — |

## Риски бизнес-уровня

| ID | Риск | Вероятность | Влияние | Меры смягчения |
| :-- | :-- | :-- | :-- | :-- |
| R-1 | Задержка запуска к началу сезона из-за расширения скоупа или технических проблем | Средняя | Высокое (потеря сезона) | Жёсткий MVP-подход; заморозка «наворотов»; еженедельный мониторинг прогресса |
| R-2 | Низкое adoption среди клиентов (клиенты продолжают писать в Telegram) | Средняя | Среднее | Активное продвижение приложения через скалодром; бонусы за первую запись через приложение |
| R-3 | Бэкенд (black box) возвращает некорректные данные или часто падает | Низкая | Высокое | Чёткий контракт API; обработка ошибок на клиенте; fallback-сообщения |
| R-4 | Нестабильный интернет в ангаре ухудшает UX | Средняя | Среднее | Retry-политика; кеширование; понятные сообщения об ошибках |
| R-5 | Клиенты не принимают правило отмены с `n` минут (не освобождается место) | Низкая | Низкое | Понятное объяснение правил в интерфейсе; push-напоминания о пороге отмены |
| R-6 | Бюджет превышен из-за неожиданных интеграционных сложностей | Средняя | Среднее | Чёткое разделение скоупа; регулярный ревью с командой разработки |
