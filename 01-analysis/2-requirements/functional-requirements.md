# Функциональные требования (FR)

> Источник: [`../1-elicitation/domain-description.md`](../1-elicitation/domain-description.md)  
> Нефункциональные требования (НФТ) в этом документе **не приводятся** — см. отдельный артефакт при необходимости.

**Легенда приоритетов:**

| Приоритет | Смысл |
|-----------|--------|
| **P0 — Must** | Без этого сезон «не взлетит» (§7) или блокирующее бизнес-правило |
| **P1 — Should** | Дополнительный сценарий MVP (§7) |
| **P2 — Could** | Заложено в домен, минимальная реализация без расширенной логики |

**Формат источника:** раздел `domain-description.md` (§) или требование брифа (R-xxx).

---

## 1. Регистрация и авторизация

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-001 | Система должна позволять регистрацию клиента по **номеру телефона**. | P0 | §4.1 |
| FR-002 | Система должна позволять регистрацию клиента через **Telegram**. | P0 | §4.1 |
| FR-003 | Система должна позволять регистрацию клиента по **email**. | P0 | §4.1 |
| FR-004 | Система должна обеспечивать **один аккаунт на одного человека**; запись на тренировку доступна только авторизованному пользователю. | P0 | §4.1, §5.1 п.1 |
| FR-005 | Система должна проверять минимальный возраст клиента — **не младше 6 лет**. | P0 | §4.1 |

---

## 2. Профиль клиента

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-006 | Система должна отображать в профиле клиента: **имя**, **телефон**, **фото**, **статус клиента**, **историю записей** (прошлые и будущие брони). | P1 | §4.1 |
| FR-007 | Система должна отображать поле **«статус клиента»** в профиле без программы лояльности (логика статуса вне MVP). | P2 | §4.1, §8 |

---

## 3. Расписание и слоты

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-010 | Система должна показывать клиенту доступные слоты тренировок **не дальше 7 дней** от текущего момента. | P0 | §4.2, §5.6 п.1, R-027 |
| FR-011 | Система должна отображать слоты по мере их появления в API (**постепенная публикация**). | P0 | §4.2, §5.6 п.2 |
| FR-012 | При отсутствии доступных слотов система должна показывать empty state: **«Пока нет доступных тренировок»**. | P0 | §5.6 п.3, R-027 |
| FR-013 | Карточка слота должна содержать: **время начала**, **длительность 1,5 ч**, **формат** (один из двух типов), **инструктора**, **зону**, **число свободных мест**, **цену тренировки**. | P0 | §4.2, §4.5, §5.1 п.3, §5.1 п.6 |
| FR-014 | Система должна отображать **два типа** форматов тренировок: болдеринг для новичков и трассы с верёвкой для опытных. | P0 | §3, §4.2 |
| FR-015 | Система должна отображать слоты со статусом **«Отменён скалодромом»** как недоступные для записи. | P1 | §4.2 |

---

## 4. Запись на тренировку

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-020 | Система должна обеспечивать **мгновенную запись** на слот при успешном ответе API. | P0 | §4.3, §5.1 п.2, §7 |
| FR-021 | Система должна **блокировать запись** для клиента с **активным долгом** до его погашения. | P0 | §4.1, §4.7, §5.5 |
| FR-022 | Система **не должна** ограничивать количество активных записей клиента вперёд. | P0 | §4.3 |
| FR-023 | Система **не должна** предоставлять лист ожидания при отсутствии свободных мест. | P0 | §4.3, §5.1 п.4, §8 |
| FR-024 | При отказе API (нет мест) система должна **корректно обработать отказ** и не создавать бронь. | P0 | §1, §5.1 п.2, R-004 |
| FR-025 | При записи клиент должен иметь возможность выбрать **прокат снаряжения** только из перечня, заданного API (скальники, страховочная система). | P0 | §3, §4.3, §5.1 |
| FR-026 | Система должна отображать **цену проката** при выборе снаряжения. | P0 | §4.3, §5.1 п.6, §7 |
| FR-027 | При нехватке прокатного снаряжения система должна показывать клиенту **предупреждение**. | P0 | §4.3, §5.1 п.5 |

---

## 5. Управление записями (отмена и прокат)

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-030 | Система должна отображать список записей клиента со статусами: **Подтверждена**, **Отменена клиентом**, **Отменена скалодромом**, **Завершена**, **No-show / долг**. | P0 | §4.3, §4.1 |
| FR-031 | Клиент должен иметь возможность **отменить запись** не позднее чем за **`n` минут** до начала; при этом **место освобождается** для других. | P0 | §4.8, §5.2, §7 |
| FR-032 | Клиент должен иметь возможность **отменить запись** позднее чем за **`n` минут** до начала; при этом место **не освобождается** для других. | P0 | §5.2, §7 |
| FR-033 | Клиент должен иметь возможность **изменить выбор проката** не позднее чем за **`n` минут** до начала тренировки. | P0 | §4.3, §4.8, §5.2 |
| FR-034 | Система должна **запретить изменение проката** позднее порога **`n` минут** (аналогично правилам отмены). | P0 | §5.2 |
| FR-035 | Значение **`n` минут** система должна получать из бэкенда (настраивается в админке, вне клиентского приложения). | P0 | §4.8 |

---

## 6. Действия скалодрома

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-040 | При отмене слота скалодромом система должна сохранять бронь со статусом **«Отменена скалодромом»** и отображать **причину отмены**. | P0 | §4.3, §5.3 п.1, R-008 |
| FR-041 | Система должна **запретить повторную запись** клиента на слот, отменённый скалодромом. | P0 | §4.3, §5.3 п.4, R-008 |
| FR-042 | При **сдвиге времени начала** слота система должна **обновить карточку записи** без отмены брони. | P1 | §5.4 |
| FR-043 | При **смене инструктора** на слоте система должна **сохранить запись** и отразить нового инструктора в карточке. | P1 | §4.5, §5.4 |

---

## 7. Абонемент / пакет занятий

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-050 | Система должна предоставлять клиенту **управление абонементом** (пакетом занятий). | P0 | §4.4, §7 |
| FR-051 | Система должна обеспечивать **связь абонемента с записями** клиента на тренировки. | P0 | §4.4, §7 |

> **Примечание:** детали типа пакета, момента списания и обязательности абонемента для записи в домене не зафиксированы (§11). FR-050 и FR-051 описывают обязательный MVP-сценарий на уровне возможностей системы.

## 8. Оценка инструкторов

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-060 | Клиент должен иметь возможность поставить инструктору оценку в **звёздах**. | P1 | §4.6, §7 |
| FR-061 | Клиент должен иметь возможность добавить к оценке **текстовый комментарий** (опционально) **от имени своего аккаунта**. | P1 | §4.6 |
| FR-062 | Оценивается **инструктор**, а не отдельная тренировка. | P1 | §4.6 |
| FR-063 | Клиент должен иметь возможность **добавить или дополнить комментарий** к оценке инструктора **в любой момент**. | P1 | §4.6 |
| FR-064 | Система **не должна** позволять клиенту изменить **звёзды** после отправки оценки (изменение — только через администратора). | P1 | §4.6 |

---

## 9. Долг за неявку

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-070 | При неявке клиента без оплаты система должна зафиксировать **долг** и отразить статус записи **No-show / долг**. | P1 | §4.3, §4.7, §5.5 |
| FR-071 | При активном долге система должна **блокировать создание новых записей** до погашения долга. | P0 | §4.7, §5.5, §7 |

## 10. Уведомления

| ID | Формулировка | Приоритет | Источник |
|----|--------------|-----------|----------|
| FR-080 | Система должна отправлять **push-напоминание** о предстоящей тренировке; **время напоминания задаёт клиент**. | P0 | §6.1, §7 |
| FR-081 | Система должна отправлять **push-уведомление** об отмене записи скалодромом. | P0 | §5.3 п.2, §6.1, R-008 |
| FR-082 | Push-уведомление об отмене скалодромом должно содержать действие **«Записаться на другой слот»**. | P0 | §5.3 п.3, §6.1 |
| FR-083 | Система должна отправлять **push-уведомление** об **изменении времени начала** тренировки. | P0 | §5.4, §6.1 |
| FR-084 | Система должна отправлять **push-уведомление** о **смене инструктора**. | P0 | §4.5, §5.4, §6.1 |
| FR-085 | Система должна отправлять **push-напоминание** об **оценке инструктора**. | P1 | §6.1 |
| FR-086 | Система должна **дублировать уведомления на email**, если email указан в профиле клиента. | P0 | §6.2 |
| FR-087 | Клиент должен иметь возможность **настроить время** push-напоминания о предстоящей тренировке. | P0 | §6.1 |

|----|--------------|-----------|----------|
